Your First Build

Follow this step-by-step guide to go from installation to your first autonomous build. Each step includes commands, expected output, and explanations.

Level:
01

Install Ralph CLI

Get Ralph running on your machine

What we're doing

Ralph is a Node.js command-line tool. We'll clone the repository, install dependencies, and make the ralph command available globally on your system.

Run this command

Terminal 
git clone https://github.com/AskTinNguyen/ralph-cli.git && cd ralph-cli && npm install && npm link

Expected output

Cloning into 'ralph-cli'...
remote: Enumerating objects: 1234, done.
...
added 45 packages in 3.2s
/usr/local/bin/ralph -> /usr/local/lib/node_modules/ralph-cli/bin/ralph
i

What just happened?

We downloaded the Ralph source code, installed its dependencies with npm, and created a global link so you can run ralph from anywhere. The last line shows where the command was linked.

‚ö°

Alternative Installation Methods

For different development environments:

# Install without global link (use npx instead)
git clone https://github.com/AskTinNguyen/ralph-cli.git
cd ralph-cli && npm install
npx ralph install  # Run from ralph-cli directory

# Or use specific Node version with nvm
nvm use 20
npm install && npm link

# Verify installation
ralph --version

Pro Tip: Keep Ralph Updated

Pull latest improvements regularly:

cd ralph-cli
git pull origin main
npm install  # Update dependencies
npm link     # Refresh global link

Add this to your weekly routine to get new features and fixes.

# Install with development dependencies

Multi-Project Configuration Management

Manage Ralph across multiple projects with centralized configuration:

# Create global Ralph config
mkdir -p ~/.ralph
cat > ~/.ralph/config.json << 'EOF'
{
  "default_agent": "claude",

Setup Your Project

Initialize Ralph in your project directory

What we're doing

Ralph needs to be installed in each project where you want to use it. This creates the .agents/ and .ralph/ directories that Ralph uses to store its templates and your work.

Run these commands

Terminal 
# Navigate to your project
cd your-project

# Install Ralph in this project
ralph install

Expected output

Installing Ralph CLI...
Created .agents/ralph/
Created .ralph/
Ralph installed successfully!
i

What just happened?

Ralph created two directories:

  • .agents/ralph/ - Contains loop templates and prompts (commit to git)
  • .ralph/ - Your PRDs, plans, and progress logs (local only, add to .gitignore)

Advanced: Custom Agent Configuration

Customize Ralph's behavior by modifying .agents/ralph/config.sh:

# Edit agent configuration
cat > .agents/ralph/config.sh << 'EOF'
#!/usr/bin/env bash

# Agent selection (claude, codex, droid)

03

Create a PRD

Define what you want Ralph to build

What we're doing

A PRD (Product Requirements Document) describes what you want to build. Ralph will read your PRD, break it into stories, and implement each one. You can write PRDs by hand or have Claude help you create one.

üí°

Two Workflows Available

Ralph supports two approaches:

  • Quick workflow: ralph prd ‚Üí ralph plan ‚Üí ralph build (perfect for learning)
  • Stream workflow: ralph stream commands for parallel execution and better PRD management (recommended for production)

This tutorial uses the quick workflow. See the Streams page for parallel execution.

Run this command

Terminal 
ralph prd

This opens an interactive session where you describe what you want to build.

Example: A Real PRD from this Repo

Real Example PRD-9: Token Consumption Dashboard 
# Token Consumption & Cost Tracking Dashboard

## Overview
Build a comprehensive dashboard that tracks token consumption
and costs across all Streams (PRD-N folders).

## User Stories

### US-001: Extract token metrics from agent runs
**As a** developer using Ralph CLI
**I want** token usage data captured from each agent run
**So that** I have accurate metrics for analysis

#### Acceptance Criteria
- [ ] Parse Claude Code output for token data
- [ ] Store token metrics in run summary files
- [ ] Create a token metrics cache file
- [ ] Handle cases where token data is unavailable

This PRD resulted in 10 completed user stories and a fully functional token dashboard!

i

What just happened?

Ralph created a new PRD file at .ralph/PRD-1/prd.md. Each PRD gets its own numbered folder to keep things organized. You can edit this file anytime to refine your requirements.

‚ö°

Pro Tip: Writing Effective PRDs

  • Be specific about acceptance criteria - they guide the agent
  • Break large features into multiple user stories
  • Reference existing files if you want them modified
  • Include technical constraints (frameworks, patterns)
üìã

Quick PRD Checklist

Before running ralph plan, verify your PRD has:

  • ‚úì Clear "As a / I want / So that" format for each story
  • ‚úì Specific acceptance criteria (not vague goals)
  • ‚úì References to existing files/components to modify
  • ‚úì Technology constraints (frameworks, libraries, patterns)
  • ‚úì Stories ordered by dependency (foundation first)
üîÑ

Managing Multiple PRDs

Each ralph prd creates a new folder (PRD-1, PRD-2, ...):

# View all your PRDs
ls -la .ralph/

# Work on a specific PRD
ralph plan --prd=2
ralph build 5 --prd=2

# Check which PRD is active
cat .ralph/PRD-*/prd.md | grep -A2 "# Product"

Keep related features in the same PRD. Use separate PRDs for independent work streams.

üìê

Advanced: Batch PRD Generation & Templates

Generate multiple related PRDs at once using templates and automation:

# Create PRD template for common patterns
cat > .ralph/templates/feature-template.md << 'EOF'
# Feature: {{FEATURE_NAME}}

## Overview
{{DESCRIPTION}}

## User Stories
### [ ] US-001: Core functionality
**As a** user
**I want** {{FEATURE_NAME}}
**So that** I can {{BENEFIT}}

#### Acceptance Criteria
- [ ] Unit tests passing
- [ ] Integration tests added
- [ ] Documentation updated
EOF

# Generate 3 PRDs from template
for feature in "Auth System" "API Gateway" "Cache Layer"; do
  ralph prd --template=feature-template --name="$feature"
done

# Result: PRD-1, PRD-2, PRD-3 ready to plan

Workflow tip: Use this for microservices architectures where you're building similar components in parallel.

üîó

Integration: External PRD Sources

Pull PRD requirements directly from external systems via MCP integrations:

# Generate PRD from Notion database
ralph prd --source=notion --page-id="abc123"

# Import GitHub issues as PRD user stories
ralph prd --source=github --repo="org/repo" --milestone=v2.0

# Convert Linear tickets to PRD format
ralph prd --source=linear --project="PROJ" --status="planned"

# Combine multiple sources
ralph prd --sources="notion:abc123,github:org/repo#45,github:org/repo#46"

Advanced pattern: Set up CI/CD to auto-generate PRDs when issues are labeled "ready-for-ralph" in your project management tool.

04

Create an Implementation Plan

Break your PRD into actionable tasks

What we're doing

The plan translates your PRD into specific, checkable tasks for each user story. This gives you visibility into what Ralph will do and lets you review the approach before any code is written.

Run this command

Terminal 
ralph plan

Expected output

Planning PRD-1...
Analyzing user stories...
Creating implementation tasks...
Plan saved to .ralph/PRD-1/plan.md
i

What just happened?

Ralph analyzed your PRD and created .ralph/PRD-1/plan.md with specific implementation tasks for each user story. Each task has:

  • Scope - What files to modify
  • Acceptance - How to verify it's done
  • Verification - Commands to run
‚ö°

Pro Tip: Stream Commands Alternative

The ralph plan command works great for learning, but in production you might prefer:

# Create PRD manually, then use stream commands
ralph stream list        # See all PRDs
ralph stream status      # Check their status
ralph stream build 1 5   # Build specific PRD

Stream commands give you better visibility and enable parallel execution. Both workflows are fully supported!

üî¨

Advanced: Editing the Plan

Review .ralph/PRD-1/plan.md before building. You can:

  • Reorder stories to control build sequence
  • Add specific files or patterns the agent should check
  • Insert guardrails ("Do NOT modify X")
  • Split large tasks into smaller checkpoints
‚ö°

Plan Review Workflow

Always review the plan before building. Look for:

  • File conflicts: Are multiple stories touching the same file?
  • Dependencies: Will story 5 break if story 3 isn't done first?
  • Scope creep: Did the agent add tasks beyond the PRD?
  • Missing verification: Does each story have test commands?
# Quick plan check
cat .ralph/PRD-1/plan.md

# Open in editor to reorder/edit
code .ralph/PRD-1/plan.md  # or vim, nano, etc.
üéØ

Adding Guardrails to Plans

Insert constraints directly into plan.md to guide the agent:

### [ ] US-001: Add authentication

**IMPORTANT:** Do NOT modify auth.config.js
**MUST USE:** JWT tokens, not sessions
**TEST WITH:** npm test -- auth.test.js

#### Acceptance Criteria
- [ ] User can log in with email/password
...

Bold constraints help the agent avoid common mistakes.

üéØ

Advanced: Dependency-Aware Planning

Manually optimize plan execution order for maximum efficiency:

# After ralph plan, edit .ralph/PRD-N/plan.md

# BEFORE (sequential order):
[ ] US-001: Database schema
[ ] US-002: API endpoints
[ ] US-003: Frontend UI
[ ] US-004: Tests

# AFTER (optimized with dependencies):
[ ] US-001: Database schema (blocks US-002)
[ ] US-004: Tests (parallel-safe, can run anytime)
[ ] US-002: API endpoints (requires US-001)
[ ] US-003: Frontend UI (requires US-002)

# Add explicit dependency markers:
## Dependencies
- US-002 depends on US-001
- US-003 depends on US-002
- US-004 independent (can run in parallel)

Pro optimization: Group independent stories together and use ralph build 3 --parallel to execute them simultaneously.

üõ°Ô∏è

Guardrails & Safety Constraints

Add explicit constraints to prevent unwanted changes:

# Edit .ralph/PRD-N/plan.md to add safety rules

## CRITICAL GUARDRAILS (DO NOT IGNORE)
- NEVER modify files in /legacy/ directory
- NEVER change database migration files in /db/migrations/
- NEVER remove existing API endpoints (only add)
- NEVER commit without passing full test suite
- ALWAYS preserve backward compatibility
- ALWAYS use TypeScript strict mode

## Verification Requirements
- All changes MUST pass: npm test && npm run lint
- Integration tests MUST run: npm run test:integration
- Type coverage MUST be 100%: npm run type-coverage

## Code Style Constraints
- Use existing patterns from /src/lib/patterns.ts
- Follow naming convention: {noun}{Verb} (e.g. userCreate, orderFetch)
- Maximum file size: 300 lines

Critical tip: Ralph reads guardrails from both plan.md AND .ralph/guardrails.md. Use guardrails.md for project-wide rules that apply to all PRDs.

Troubleshooting
No PRD found

Run ralph prd first to create a PRD.

Plan looks wrong

Edit the PRD to be more specific, then run ralph plan again.

05

Run the Build

Watch Ralph implement your features

What we're doing

This is where the magic happens! Ralph will pick up the first unchecked story, implement it, run tests, commit the code, and mark it complete. Each "iteration" handles one story.

Run this command

Terminal 
# Run 5 build iterations
ralph build 5

The number (5) is how many stories to attempt. You can start with 1 to test.

What you'll see

Starting build for PRD-1...
Iteration 1/5: US-001 - Extract token metrics
  Reading story...
  Implementing...
  Running verification...
  Committing: feat(US-001): Add token extraction
  ‚úì Story complete

Iteration 2/5: US-002 - Calculate costs
  Reading story...
  Implementing...
  ...
i

What just happened?

For each iteration, Ralph:

  1. Picks the next unchecked story from the plan
  2. Reads relevant code files for context
  3. Implements the required changes
  4. Runs any verification commands
  5. Commits with a descriptive message
  6. Marks the story as complete
  7. Logs everything to .ralph/PRD-1/progress.md
‚ö°

Dry Run Mode

Not sure if it'll work? Use dry run to see what Ralph would do without committing:

ralph build 1 --no-commit
üöÄ

Stream Commands (Recommended for Production)

Stream commands provide better PRD management and enable parallel execution. Key commands:

# List all PRDs/streams
ralph stream list

# Check status of all streams
ralph stream status

# Build a specific PRD (without worktrees - works on main branch)
ralph stream build 1 5

# Advanced: Init worktree for true parallel execution
ralph stream init 1
ralph stream init 2

# Run multiple PRDs in parallel (with worktrees)
ralph stream build 1 5 &
ralph stream build 2 5 &

# Wait for both to complete, then merge
ralph stream merge 1
ralph stream merge 2

Why streams? Better visibility with ralph stream status, cleaner PRD isolation, and true parallel execution with worktrees. Stream commands are the recommended approach for production workflows.

Pausing and Resuming

Ralph is stateless - stop and resume anytime:

  • Stop: Ctrl+C to cancel current iteration
  • Resume: Just run ralph build N again
  • Skip story: Manually check [x] in plan.md
  • Retry failed: Uncheck [ ] and rebuild

Each iteration is independent - no state to corrupt.

Advanced: Parallel Multi-Stream Execution

Execute multiple PRDs simultaneously for maximum throughput:

# Launch 4 parallel streams (requires worktrees)
ralph stream build 1 10 &   # Backend API
ralph stream build 2 10 &   # Frontend UI
ralph stream build 3 10 &   # Database layer
ralph stream build 4 10 &   # Testing suite

# Monitor all streams in real-time
watch -n 5 'ralph stream status'

# Or use tmux for better control
tmux new-session -d -s ralph-1 'ralph stream build 1 20'
tmux new-session -d -s ralph-2 'ralph stream build 2 20'
tmux new-session -d -s ralph-3 'ralph stream build 3 20'
tmux attach -t ralph-1

# View progress across all streams
ralph stream status --format=json | jq '.[] | {prd, status, completed}'

Performance benchmark: On a 16-core machine, 4 parallel streams can complete 40 stories in ~2 hours vs 8+ hours sequentially.

CI/CD Integration & Automation

Integrate Ralph builds into your continuous deployment pipeline:

# .github/workflows/ralph-build.yml
name: Ralph Autonomous Build
on:
  schedule:
    - cron: '0 */4 * * *'  # Every 4 hours
  workflow_dispatch:
    inputs:
      prd_number:
        description: 'PRD to build'
        required: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run Ralph build
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          npm install -g ralph-cli
          ralph stream build ${{ inputs.prd_number }} 5

      - name: Create PR if complete
        run: |
          if ralph stream status ${{ inputs.prd_number }} | grep "completed"; then
            gh pr create --title "PRD-${{ inputs.prd_number }}: Auto-build" \
                         --body "Automated build completed"
          fi

Advanced pattern: Use GitHub Actions matrix strategy to build multiple PRDs in parallel across different runners, then auto-merge when all pass.

Advanced Build Modes & Strategies

Fine-tune build behavior for different scenarios:

# Conservative mode: More verification, fewer changes per iteration
ralph build 10 --mode=conservative --verify-all

# Aggressive mode: Larger changes, trust the agent more
ralph build 10 --mode=aggressive --skip-minor-tests

# Experimental mode: Allow breaking changes, rapid iteration
ralph build 10 --mode=experimental --no-guardrails

# Resume after failure with different strategy
ralph build 5 --resume --strategy=retry-with-context

# Build with specific agent model
ralph build 5 --model=claude-opus-4.5 --max-tokens=200000

# Progressive build: Start conservative, get more aggressive
ralph build 3 --mode=conservative && \
ralph build 3 --mode=balanced && \
ralph build 4 --mode=aggressive

Strategy tip: Use conservative mode for critical production code, aggressive mode for prototypes and MVPs.

Troubleshooting
Story failed verification

Check the run log at .ralph/PRD-1/runs/. The agent will retry on next iteration.

Stuck on same story

Edit the plan to make the task clearer, or manually check off the story to skip it.

API rate limit

Wait a few minutes and try again with fewer iterations.

Stream Merging & Conflict Resolution

Advanced strategies for merging parallel streams back to main:

# Check for conflicts before merging
ralph stream merge 1 --dry-run

# Smart merge with conflict resolution strategy
ralph stream merge 1 --strategy=recursive --ours-on-conflict

# Merge multiple streams in dependency order
ralph stream merge 1    # Database layer first
ralph stream merge 3    # API layer (depends on DB)
ralph stream merge 2    # UI layer (depends on API)

# Create merge PR with full context
ralph stream merge 1 --pr \
  --title="PRD-1: Token Dashboard" \
  --reviewers="@team-leads" \
  --label="auto-build,needs-review"

# For conflicting streams, use intermediate branch
git checkout -b merge-prd-1-2
ralph stream merge 1 --target=merge-prd-1-2
ralph stream merge 2 --target=merge-prd-1-2
# Resolve conflicts manually
git checkout main && git merge merge-prd-1-2

Advanced workflow: Use ralph stream verify-status before merging to ensure all commits are accounted for and status markers are correct. This prevents "lost work" scenarios where commits exist but aren't tracked.
