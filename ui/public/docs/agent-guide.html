‚ö° Quick Start for Agents

üö® CRITICAL RULES
  • ALWAYS use --headless when running ralph prd
  • NEVER auto-merge - builds require explicit human approval
  • Git commits = source of truth, not checkboxes
  • Exit after completion - let humans handle merges/PRs
‚úÖ MOST COMMON COMMANDS
ralph prd "description" --headless
ralph stream status
ralph stream build N [iterations]
git log --grep="PRD-N" # verify status
ralph error RALPH-XXX # lookup errors
Agent workflow: (1) Generate PRD with --headless ‚Üí (2) Check status ‚Üí (3) Run build ‚Üí (4) Verify via git log ‚Üí (5) Exit with <promise>COMPLETE</promise>
‚ö†Ô∏è CRITICAL: Nested Agent Interaction Warning
When Claude Code (or another AI agent) executes ralph prd, it invokes a nested agent. Without --headless, both agents try to interact with the same TTY, causing:
  • Deadlocks - Both agents waiting for input
  • TTY conflicts - Overlapping I/O streams
  • Process hangs - Commands never complete
  • Unpredictable output - Garbled or missing responses
‚úÖ ALWAYS USE:
ralph prd "description" --headless
‚ùå NEVER USE:
ralph prd "description"
(causes nested interaction)
üö® CRITICAL: Manual Merge Policy
Ralph NEVER auto-merges builds to main. This is a core safety guarantee.
‚ùå NEVER DO (Prohibited):
  • Run ralph stream merge commands
  • Create pull requests automatically
  • Push branches to remote automatically
  • Attempt any form of automatic merge
  • Suggest that you will merge on completion
‚úÖ CORRECT AGENT WORKFLOW:
  1. Execute the assigned stories
  2. Output <promise>COMPLETE</promise> signal
  3. Exit - let the human handle merge/PR creation
‚ÑπÔ∏è USER WORKFLOW (What Happens Next):
  1. User reviews commits: git log, git diff
  2. User validates build: npm test, npm run build
  3. User manually triggers merge: ralph stream merge N
  4. User confirms merge at interactive prompt (or uses --yes flag)

Why this matters: Merging requires explicit human validation. Build completion does NOT imply merge approval. This separation ensures code quality and prevents unreviewed changes from reaching main.

Common Task Patterns

Pattern 1: Quick Single-PRD Build

# User: "Build a new feature X"
# IMPORTANT: Agents must use --headless to avoid nested interaction
ralph prd "Feature X description" --headless
# Wait for PRD generation (creates .ralph/PRD-N/prd.md)
ralph plan
# Wait for plan generation (creates .ralph/PRD-N/plan.md)
ralph build 5
# Monitor output, report completion

Pattern 2: Stream-Based Build (Recommended)

# User: "Build feature X using streams"
# IMPORTANT: Agents must use --headless to avoid nested interaction
ralph prd "Feature X description" --headless
# Note the PRD number from output (e.g., "PRD-3")
ralph stream status
# Verify PRD created, status shows "ready"
ralph stream build 3 5
# Monitor progress, check status periodically

Pattern 3: Parallel Execution (Advanced)

# User: "Build PRD-1 and PRD-2 in parallel"
# Option A: Without worktrees (simpler, commits to main)
ralph stream build 1 5 --no-worktree &
ralph stream build 2 5 --no-worktree &
wait  # Wait for both to complete

# Option B: With worktrees (true isolation, requires merge)
ralph stream init 1
ralph stream init 2
ralph stream build 1 5 &
ralph stream build 2 5 &
wait
ralph stream merge 1
ralph stream merge 2

Pattern 4: Status Check & Monitoring

# User: "What's the status of my PRDs?"
ralph stream status
# Shows: ready, running, completed, merged, in_progress

# Check specific PRD progress
cat .ralph/PRD-N/progress.md

# View recent commits
git log --oneline --grep="PRD-N" -10

PRD Command Modes

Agent Rule: Always Use --headless
ralph prd "description" --headless

Why? Prevents nested agent interaction, TTY conflicts, and process hangs.
When? Claude Code agent, UI server, CI/CD, background jobs, any non-terminal context.
Quick Examples
# Agent execution (REQUIRED)
ralph prd "Add dashboard feature" --headless

# Human terminal (optional headless)
ralph prd "Build auth system"

Critical Rules

‚ùå DO NOT
  • Run ralph prd without --headless (causes nested agent interaction)
  • Run build and stream build simultaneously on same PRD
  • Edit .ralph/PRD-N/ while build running
  • Use --force without user permission
  • Merge before status is "completed"/"merged"
  • Trust checkboxes as proof of completion (git log is source of truth)
‚úÖ DO
  • Always use ralph prd "..." --headless when executing as agent
  • Check ralph stream status before builds
  • Use --no-worktree for simple builds
  • Verify git commits: git log --grep="PRD-N" (source of truth)
  • Read .ralph/PRD-N/progress.md for iteration details
  • Launch ralph ui for monitoring
  • Reference error codes when reporting failures

Status Codes

Git Commits = Source of Truth
  • Git history is authoritative, not checkboxes
  • Two workflows: merged (worktree ‚Üí PR) or completed (direct-to-main)
  • Auto-correction: Missing .completed markers auto-created when git shows commits
  • Checkboxes are hints: Used by agents during work, not for status determination
Status Meaning Action
ready plan.md exists, no progress yet ralph stream build N
running Lock file exists with active PID Wait or check progress.md
in_progress progress.md exists but no commits Resume build (work not committed)
completed Commits on main (direct-to-main workflow) Done. Review git log --grep="PRD-N"
merged Branch merged to main (worktree workflow) Done. PR merged
not_found PRD directory doesn't exist ralph prd "..." --headless

Status Verification Commands

# Check if PRD has commits on main (source of truth)
git log --oneline --grep="PRD-N"

# Verify specific commits from progress.md
grep "Commit:" .ralph/PRD-N/progress.md

# Auto-scan and fix all stale status markers
ralph stream verify-status

# Manually mark as completed (if commits exist on main)
ralph stream mark-completed N

# Remove completion marker
ralph stream unmark-completed N

Common Status Issues

  • Checkbox marked but no commits ‚Üí Status = "in_progress" (work not committed yet)
  • Commits exist but shows "ready" ‚Üí Run ralph stream verify-status to auto-correct
  • Distinguish worktree vs direct builds ‚Üí Check for .merged vs .completed marker files
  • Direct-to-main PRD shows wrong status ‚Üí get_stream_status() will auto-create .completed marker on first check

Quick Troubleshooting

Build hangs or fails

# Check if lock exists
ls -la .ralph/locks/

# Remove stale lock (only if process truly dead)
rm .ralph/locks/PRD-N.lock

# Check error logs
cat .ralph/PRD-N/errors.log
tail -n 50 .ralph/PRD-N/runs/run-*.log

Status shows wrong state

# Verify status with git history (source of truth)
git log --all --oneline --grep="PRD-N"

# Auto-fix stale status markers
ralph stream verify-status

# Manually mark as completed (if commits exist)
ralph stream mark-completed N

Merge conflicts

# If merge fails, check worktree branch
cd .ralph/worktrees/PRD-N/
git status
git diff main

# Return to main repo
cd ../../..

# User must resolve conflicts manually

# Interact

7-Step Testing Checklist

  1. Navigate: agent-browser open http://localhost:3000
  2. Snapshot: agent-browser snapshot -i
  3. Verify visible: agent-browser is visible @e1
  4. Test interactions: agent-browser click @e2
  5. Check console: agent-browser console && agent-browser errors
  6. Validate data: agent-browser get text @e1

Writing Quality Specifications (Tech-Agnostic)

Ralph's Design Philosophy
Ralph works across Python, JavaScript, Go, Rust, Java, etc. All improvements are:
‚úÖ Tech-agnostic (no prescriptive stacks)
‚úÖ Auto-detect project context (package.json, Cargo.toml, etc.)
‚úÖ Configurable via optional .ralph/standards.md
‚úÖ PRD = WHAT (requirements), Plan = HOW (implementation)

Six Core Areas

Ralph PRDs should include: Commands, Testing, Project Structure, Standards, Boundaries
Note: Code patterns belong in Plans, not PRDs (WHAT vs HOW)

Commands (Project-Specific Auto-Detection)

‚ùå Bad: Generic placeholders 
"Run the tests"
‚úÖ Good: Detect from project 
# JavaScript: npm test -- --testPathPattern=auth
# Python: pytest tests/test_auth.py
# Rust: cargo test auth
# Go: go test ./auth

Three-Tier Boundaries (Universal)

Tier Permission Examples
‚úÖ Always No permission needed Run tests, modify feature files
‚ö†Ô∏è Ask Requires approval DB schema, new deps, shared utils
üö´ Never Prohibited Commit secrets, delete tests, skip checks

Tech-Agnostic Principles

  • Auto-detect tech stack: Read project files (package.json, Cargo.toml, go.mod, requirements.txt)
  • Follow existing patterns: Inspect codebase, don't invent generic examples
  • Use project's commands: Check README, Makefile, package scripts
  • PRD = WHAT, Plan = HOW: Keep PRDs high-level, Plans implementation-focused

Common Pitfalls

  • ‚ùå Assuming tech stack: "Use TypeScript interfaces" ‚Üí ‚úÖ "Define data types per project language"
  • ‚ùå Generic examples: "Follow async patterns" ‚Üí ‚úÖ "Copy error handling from src/services/example.py"
  • ‚ùå Vague commands: "Run tests" ‚Üí ‚úÖ "pytest tests/feature.py -v"

Code Patterns (Plans Only, Not PRDs)

When creating implementation plans:
1. Read the project codebase to find existing patterns
2. Copy 2-3 concrete examples (error handling, validation, testing)
3. Reference actual files (e.g., "Found in: src/services/example.js")
4. Never invent generic examples - always project-specific

Standards File (Optional)

Create .ralph/standards.md for project-specific conventions:
‚Ä¢ Git workflow (branch naming, commit format)
‚Ä¢ Code quality requirements (pre-commit checklist)
‚Ä¢ Custom boundaries (project-specific Always/Ask/Never)

Example: cp .ralph/standards.md.example .ralph/standards.md
