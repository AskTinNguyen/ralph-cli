Documentation / Best Practices

Tips, Tricks & Best Practices

Curated wisdom from experienced Ralph users to help you work more effectively and avoid common pitfalls.

Pro Tips

Actionable advice to level up your Ralph workflow.

1

Start with a Detailed PRD

The more specific your PRD, the better Ralph performs. Include concrete acceptance criteria with measurable outcomes rather than vague goals.

Example

Better: "Add user authentication with email/password, JWT tokens, and session expiry of 24 hours"

Worse: "Add login feature"

2

Use Streams for Independent Features

When you have multiple PRDs that touch different parts of your codebase, run them in parallel using streams. This can speed up development by 3-4x.

Use Case

Run frontend redesign (PRD-1) and backend API refactor (PRD-2) simultaneously since they touch different files.

3

Keep Stories Small and Focused

Break large features into user stories that can be completed in 1-3 build iterations. Smaller stories are easier to verify and debug.

Example Breakdown

Instead of "Implement complete user management", split into:

  • • US-001: User registration endpoint
  • • US-002: Login with JWT
  • • US-003: Password reset flow
  • • US-004: Profile update endpoint
4

Review Progress Logs After Each Run

The progress.md file documents what was implemented and what was learned. Review it after each run to understand what Ralph accomplished and catch any issues early.

5

Specify Verification Commands in Acceptance Criteria

Include exact commands Ralph should run to verify the story is complete. This ensures consistent quality and makes debugging easier.

- [ ] Tests pass: npm test -- user.test.ts
- [ ] Build succeeds: npm run build
- [ ] No type errors: npx tsc --noEmit
6

Use Dry Runs to Preview Changes

Test your PRD and plan with ralph build 1 --no-commit to see what Ralph will do without committing changes. Great for validating your story structure.

7

Use Descriptive Story IDs

Story IDs like US-001, US-002 make it easy to track work in git commits and progress logs. They also help when multiple team members are working on the same project.

8

Start with More Iterations Than You Think

It's better to run ralph build 10 and finish early than run ralph build 3 and need to restart. Unused iterations don't cost anything.

9

Leverage MCP Servers for Context

Connect Ralph to Notion, Slack, or GitHub via MCP servers to pull in requirements, post updates, or search existing code. This reduces manual context switching.

10

Check Stream Status Before Merging

Always run ralph stream status before merging to verify the build completed successfully and there are no lock conflicts.

Common Mistakes & How to Fix Them

Learn from others' experiences and avoid these pitfalls.

Mistake #1: Vague Acceptance Criteria
Problem

Writing acceptance criteria like "Feature works correctly" or "Code is clean" doesn't give Ralph clear targets.

Fix

Use specific, testable criteria with measurable outcomes:

  • API returns 200 status for valid requests
  • Response time is under 500ms
  • All unit tests pass
Mistake #2: Running Streams Concurrently on Same Files
Problem

Starting two streams that modify the same files creates merge conflicts.

Fix

Use ralph stream status to check which files are being modified, or run sequential builds instead of streams.

Safe: PRD-1 (frontend) + PRD-2 (backend) in parallel

Unsafe: PRD-1 (user auth) + PRD-2 (user profile) - both touch user.ts

Mistake #3: Not Reading Guardrails Before Building
Problem

Repeating the same errors that were already documented in guardrails.md.

Fix

Ralph reads guardrails automatically, but you should review them before writing your PRD to avoid known issues.

Mistake #4: Overwriting Plans by Running ralph plan Multiple Times
Problem

In older versions, running ralph plan again would overwrite your existing plan.

Fix

Current version auto-increments to PRD-N folders. Each ralph plan creates a new isolated plan.

Mistake #5: Too Many Stories in One PRD
Problem

Creating a PRD with 20+ stories makes it hard to track progress and increases the chance of merge conflicts.

Fix

Split large initiatives into multiple PRDs (3-7 stories each) and run them sequentially or as independent streams.

Mistake #6: Skipping Browser Testing for UI Changes
Problem

Testing UI features with curl or API calls doesn't catch rendering issues.

Fix

Use the chromemcp MCP server for browser automation. Include browser verification commands in your acceptance criteria.

Workflow Optimization

Choose the right approach for your situation.

When to Use Streams vs Sequential Builds

Question: Do you have multiple PRDs to implement?
↓ Yes
Question: Do the PRDs modify different files/components?
↓ Yes
Use Streams: ralph stream init N && ralph stream build N 5
↓ No (same files)
Use Sequential: ralph build 5 --prd=1 then ralph build 5 --prd=2

Stream Benefits:

  • 3-4x faster for independent PRDs
  • Isolated git worktrees prevent conflicts
  • Can merge PRDs in any order

Stream Drawbacks:

  • Requires more disk space (separate worktrees)
  • Merge conflicts if PRDs overlap
  • More complex to debug

Optimal Iteration Sizing

The number of iterations depends on story complexity:

  • Simple stories (1-2 files): 3-5 iterations
  • Medium stories (multiple files, some tests): 5-10 iterations
  • Complex stories (architecture changes, integrations): 10-15 iterations

Ralph will stop early if the story completes before max iterations, so err on the side of more.

Team Collaboration Strategies

For teams:

  • Each developer creates their own PRD-N folder
  • Use descriptive commit messages with PRD and story IDs
  • Review progress.md before merging to understand changes
  • Use ralph stream status to see who's working on what

CI/CD Integration

Run Ralph in CI/CD pipelines for automated PRD execution:

GitHub Actions Example 
- name: Run Ralph Build
  run: |
    ralph build 5 --prd=${{ github.event.inputs.prd_number }}
    git push origin main

Use cases:

  • Nightly builds of pending PRDs
  • Automated refactoring tasks
  • Documentation updates

PRD Writing Best Practices

Craft PRDs that lead to successful implementations.

Structure Your PRD Effectively

A well-structured PRD should include:

  • Overview: High-level description of what you're building and why
  • User Stories: Specific features broken into implementable chunks
  • Acceptance Criteria: Measurable outcomes for each story
  • Technical Considerations: Architecture notes, dependencies, constraints

Good vs Bad User Stories

Good Story

US-001: User Registration API

As a mobile app developer
I want a POST /api/register endpoint
So that users can create accounts

Acceptance Criteria:

  • Accepts email and password in request body
  • Returns 201 status with user ID and JWT token
  • Returns 400 for invalid email format
  • Returns 409 for duplicate email
  • Passwords are hashed with bcrypt
Bad Story

US-001: Add User System

As a user
I want to use the app
So that I can do things

Acceptance Criteria:

  • Users can register
  • Users can login
  • It works correctly

Key differences:

  • Good story has specific persona (mobile app developer)
  • Good story defines exact API contract
  • Good story has testable, measurable criteria
  • Bad story is too vague and broad

Include Context and Constraints

Help Ralph make better decisions by including:

  • Existing architecture: "Use the existing AuthService pattern"
  • Technology choices: "Use Hono for the API, not Express"
  • Performance requirements: "API response time must be under 200ms"
  • Security considerations: "Follow OWASP best practices for input validation"

Order Stories by Dependency

Sequence stories so each builds on the previous:

Good order
  1. US-001: Database schema and models
  2. US-002: API endpoints
  3. US-003: Frontend components
  4. US-004: Integration tests

This prevents Ralph from trying to build components before their dependencies exist.

Integration Patterns

Connect Ralph to your existing tools and workflows.

Git Hooks for Automated PRD Updates

Use git hooks to keep PRD status synchronized with your repository:

post-commit hook (.git/hooks/post-commit) 
#!/bin/bash
# Auto-update PRD status after commit
if git log -1 --pretty=%B | grep -q "PRD-"; then
  echo "PRD commit detected, updating status..."
  ralph stream verify-status
fi

Notion Integration for Requirements

Pull requirements directly from Notion databases:

  • Store user stories in a Notion database
  • Use the Notion MCP server to fetch stories
  • Generate PRD content from Notion properties
Setup
  1. Add NOTION_API_KEY to your environment
  2. Configure Notion database ID in your PRD
  3. Ralph can read stories during planning phase

Slack Notifications for Build Updates

Get real-time updates when builds complete:

Custom skill (skills/notify/skill.sh) 
# Post to Slack when build completes
curl -X POST $SLACK_WEBHOOK_URL \
  -d "{ \"text\": \"PRD-$PRD_NUM build completed!\" }"

GitHub Issues Integration

Sync Ralph stories with GitHub Issues:

  • Create GitHub issues from PRD user stories
  • Link commits to issues using issue numbers
  • Auto-close issues when stories complete
Commit message format
feat(PRD-1): US-003 - Add user dashboard (fixes #42)

Monorepo Strategy

Use Ralph in monorepos by scoping PRDs to specific packages:

  • Create separate .ralph directories per package
  • Or use path prefixes in story descriptions
  • Run builds with package-specific test commands
Package-scoped PRD
packages/api/.ralph/PRD-1/prd.md packages/web/.ralph/PRD-2/prd.md
